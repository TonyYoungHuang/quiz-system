# 微信题库小程序开发任务清单

## 1. 目标定义

本项目的业务目标是：

- 在虚拟商品店铺售卖小程序刷题激活码
- 用户购买后输入激活码，开通指定科目刷题权限
- 后台可持续新增科目
- 后台可批量导入题库
- 每个科目支持按不同训练模块刷题
- 训练模块至少覆盖：`历年真题`、`考前冲刺`、`模拟练习`

---

## 2. 开发原则

- 每次开发只做一个独立任务
- 每个任务必须可单独验收
- 优先保证“卖码-激活-刷题-继续扩科目”的主链路
- 先统一架构，再扩功能，避免云开发版和 Node 版继续分叉

---

## 3. 建议主线

建议将当前项目收口为：

- 小程序前端：微信原生小程序
- 主后端：微信云开发版
- 管理端：云开发管理后台

原因：

- 你当前的业务核心是卖激活码和持续运营题库，云开发足够支撑 MVP
- 仓库里虽然有 Node 版增强能力，但现在已出现双后端并行，继续叠加会提高维护成本
- 先把云开发主线补齐，再决定是否保留 Node 版做增强版或迁移版

---

## 4. 模块化开发任务

下面的任务已经按“一个任务就是一次独立开发”的方式拆开。

### T01. 架构收口与环境统一

- 目标：统一当前系统到底走云开发还是走 Node，先消除双后端混用
- 当前状态：已有基础，但未完成
- 主要工作：
- 统一小程序请求策略，确定是否默认走云函数
- 整理 `baseUrl`、云环境 ID、环境切换方式
- 明确管理后台调用方式，避免一部分走云函数、一部分走 HTTP API
- 更新项目说明文档
- 交付结果：
- 前端只保留一套明确可运行的数据链路
- 管理后台只保留一套明确可运行的数据链路
- 验收标准：
- 新人打开项目后，能明确知道该部署哪一套后端
- 不再出现“客户端走 A，后台部分功能走 B”的情况

### T02. 微信用户身份与权限体系

- 目标：把当前本地临时 `userId` 改成真实微信身份
- 当前状态：待开发
- 主要工作：
- 接入微信登录身份
- 获取并保存用户唯一标识
- 让权限记录绑定真实用户，而不是本地缓存生成的临时 ID
- 处理换手机、清缓存后的权限恢复
- 交付结果：
- 用户重新登录后仍可找回已激活科目
- 验收标准：
- 同一微信用户更换设备后，仍能读取自己的已激活权限

### T03. 激活码售卖与核销闭环

- 目标：打通店铺售卖激活码后的开通流程
- 当前状态：已有基础，可继续完善
- 主要工作：
- 保留现有激活码输入与核销流程
- 完善激活码生成规则、来源字段、使用记录
- 增加激活码状态追踪：未使用、已使用、使用人、使用时间
- 增加异常场景处理：重复使用、科目不匹配、权限已存在
- 交付结果：
- 店铺售卖后可批量发码，用户输入即可开通对应科目
- 验收标准：
- 单个激活码只能使用一次
- 用户输入正确激活码后，立即获得目标科目权限

### T04. 科目管理模块

- 目标：后台可持续新增、编辑、排序、上下架科目
- 当前状态：已有基础
- 主要工作：
- 完善科目新增、编辑、删除、排序
- 增加科目状态管理：启用、停用
- 增加科目图标、分类、说明字段
- 补齐删除科目后的关联清理逻辑
- 交付结果：
- 后台可独立维护科目，不改前端代码也能新增题库入口
- 验收标准：
- 后台新增一个科目后，小程序首页自动显示

### T05. 题库批量导入模块

- 目标：后台可按科目批量导入题目
- 当前状态：已有基础，可加强
- 主要工作：
- 固化 CSV/文本导入格式
- 导入时做题型校验、答案校验、重复校验
- 生成导入结果报告：成功数、失败数、失败原因
- 支持大批量题库分批导入
- 交付结果：
- 运营人员可不写代码，直接导入新科目的题库
- 验收标准：
- 任意一个新科目都能通过后台完成批量导题

### T06. 题目模型与刷题引擎

- 目标：统一题目结构，保证答题页稳定可用
- 当前状态：已有基础，可继续打磨
- 主要工作：
- 明确支持的题型范围
- 统一题干、选项、答案、解析、图片、公式等字段结构
- 完善答题页的提交、判分、跳题、答题卡逻辑
- 明确短答题、计算题、材料题的判分规则
- 交付结果：
- 各题型都能稳定展示和答题
- 验收标准：
- 至少单选、多选、判断必须稳定可用
- 新增题型必须有明确展示规则和判分规则

### T07. 专题训练模块

- 目标：每个科目可按专题分类刷题
- 当前状态：已有页面和部分接口基础，未闭环
- 适用场景：
- 考前冲刺
- 高频考点
- 章节训练
- 模块训练
- 主要工作：
- 建立专题数据表
- 支持专题树结构或一级/二级专题
- 支持将题目绑定到专题
- 小程序可按专题进入答题
- 后台可新增、编辑、删除专题
- 交付结果：
- 一个科目下可拆出多个专题训练入口
- 验收标准：
- 用户点击某个专题，只刷该专题下的题

### T08. 真题试卷模块

- 目标：每个科目支持按年份或试卷组织刷题
- 当前状态：已有页面和部分接口基础，未闭环
- 适用场景：
- 2025 年真题
- 2024 年真题
- 模拟卷 1
- 模拟卷 2
- 主要工作：
- 建立试卷数据表
- 支持试卷标题、年份、排序
- 支持将题目绑定到试卷
- 小程序可按试卷进入答题
- 后台可新增、编辑、删除试卷
- 交付结果：
- 用户可以直接选择某一套真题或模拟卷开刷
- 验收标准：
- 用户点击“2025 年真题”时，只出现该卷绑定的题

### T09. 科目首页训练入口模块

- 目标：把“科目 -> 专题训练 / 真题试卷 / 普通刷题”入口设计清楚
- 当前状态：已有初版入口
- 主要工作：
- 首页展示科目列表
- 进入科目后展示训练方式入口
- 明确入口文案与层级结构
- 优化未激活、已激活两种状态下的跳转逻辑
- 交付结果：
- 用户在一个科目内能清楚选择刷题方式
- 验收标准：
- 不需要学习成本，用户一眼能看懂如何进入真题、冲刺、模拟练习

### T10. 错题本与收藏夹模块

- 目标：增强复习闭环，提高复购和留存
- 当前状态：占位页，待开发
- 主要工作：
- 记录用户错题
- 支持用户收藏题目
- 提供“只看错题”“只看收藏题”模式
- 支持清空、移除、再次练习
- 交付结果：
- 用户可持续复习重点题
- 验收标准：
- 用户答错后，该题能出现在错题本中
- 用户收藏后，该题能出现在收藏夹中

### T11. 答题记录与学习统计模块

- 目标：让用户能看到自己的刷题进度，提升使用感知
- 当前状态：待开发
- 主要工作：
- 保存答题记录
- 统计最近练习次数、正确率、最近学习时间
- 统计每个科目的练习数据
- 在“我的”页面展示学习数据
- 交付结果：
- 用户能看到自己的学习轨迹
- 验收标准：
- 完成一次练习后，用户中心能看到记录和统计

### T12. 管理后台增强模块

- 目标：把后台补成真正可运营的题库管理系统
- 当前状态：基础能力已有，扩展能力未统一
- 主要工作：
- 补齐专题管理
- 补齐试卷管理
- 补齐导入任务记录
- 提供题目绑定专题/试卷的后台操作
- 提供激活码批量生成、复制、筛选、删除
- 交付结果：
- 运营人员可独立维护科目、题库、专题、试卷、激活码
- 验收标准：
- 不改代码即可完成新增科目和完整题库上线

### T13. 运营上线与部署模块

- 目标：把系统从开发状态推进到可运营状态
- 当前状态：待收尾
- 主要工作：
- 补齐云函数部署说明
- 补齐数据库初始化说明
- 补齐后台访问说明
- 配置正式环境参数
- 梳理店铺发码与售后处理流程
- 交付结果：
- 项目可被稳定部署、交接、复用
- 验收标准：
- 按文档从零部署一次后，能完成发码、激活、刷题全链路

1. `T01 架构收口与环境统一`

3. `T03 激活码售卖与核销闭环`
4. `T04 科目管理模块`
5. `T05 题库批量导入模块`

## 8. 单次开发的执行模板

后续每次开发可以固定按这个模板推进：

1. 明确本次只做哪个任务
2. 写清任务目标和不做范围
3. 先补数据结构
4. 再补后端接口
5. 再补前端页面
6. 最后补验收用例和文档

建议每次开发结束后，都产出以下结果：

- 改动文件清单
- 本次完成内容
- 未完成内容
- 验收步骤
- 下一个任务建议

---

## 9. 你现在最应该先做的事

如果只给一个建议，优先做：

- `T01 架构收口与环境统一`

因为当前项目最大的风险不是功能少，而是：

- 文档说云开发是主线
- 小程序实际默认优先走 HTTP API
- 后台基础功能走云函数
- 专题/试卷/导入任务又走 Node 接口

这个问题不先解决，后面每增加一个功能，维护成本都会继续上升。
